# IDOL CROSSING ゼロベース再設計メモ（実験案）

## 0. 前提

- これは「いまゼロから自由に作り直せるなら」を想定した再設計メモ
- 現行の強み（ニュース / イマキテ / ナンダッテ / バズッタラ / 横断検索）は維持しつつ、構造を再定義する
- 目的は「推し活の情報導線」と「参加体験」を同じプロダクト内で連続化すること

## 1. 再設計の結論（先に要点）

- サービスを4機能の寄せ集めではなく、**1つの「グループタイムライン基盤」**として再構成する
- すべての情報を `Group` を中心に正規化し、ニュース・ランキング・SNS反響・投票を同一時系列で扱う
- UIは「ページ単位」より「タスク単位（知る / 比べる / 参加する / 振り返る）」に再編する
- 運用は「手動更新前提」から「イベント駆動 + 夜間バッチ + 品質監視」へ切り替える

## 2. プロダクト設計（ユーザー体験）

### 2.1 情報設計の軸

- 主キーは `Group`
- 全データに `group_id` と `occurred_at` を持たせ、比較可能にする
- 入口は複数でも、遷移先は必ず `Group Hub`（グループ統合画面）に収束させる

### 2.2 画面構成（新）

- `Home`: 今日の動き（主要ニュース、急上昇、話題投稿、投票変動）
- `Explore`: 横断検索 + フィルタ（期間、カテゴリ、SNS、勢い）
- `Group Hub`: 1グループの全情報を時系列で集約
- `Participate`: ナンダッテ投票、注目テーマ、投票トレンド
- `My Page`: フォロー、投票履歴、閲覧履歴、通知設定

### 2.3 体験フロー

- 知る: ニュース・バズを確認
- 比べる: イマキテで日次/週次比較
- 参加する: ナンダッテに投票・コメント
- 振り返る: My Pageで自分の行動と推しの変化を追う

## 3. ドメインモデル（データ設計）

### 3.1 コアエンティティ

- `groups`（グループ）
- `articles`（ニュース）
- `social_posts`（SNS話題）
- `rank_snapshots`（日次/週次ランキング）
- `vote_topics`（投票テーマ）
- `votes`（投票ログ）
- `users` / `profiles`
- `group_events`（統合イベントテーブル）

### 3.2 重要方針

- UI向け集約は `group_events` に投影（CQRSのRead Model）
- 生データ（source of truth）と表示最適化データ（read model）を分離
- 検索は全文索引 + 同義語辞書（グループ名揺れ、略称、旧名対応）

## 4. 技術アーキテクチャ（ゼロベース案）

### 4.1 アプリ構成

- Frontend: Next.js App Router（SSR + ISR + Partial Prerendering）
- API/BFF: Next.js Route Handlers または独立Node API
- DB: PostgreSQL（Supabase可）
- Search: PostgreSQL FTS開始、必要時にMeilisearch/Typesenseへ拡張
- Queue/Job: pg-boss or Cloud Tasks（集計・再計算・同期）
- Analytics: PostHog + BigQuery（またはClickHouse）

### 4.2 データパイプライン

- Ingest層: WP/SNS/外部APIの取得コネクタ
- Normalize層: group紐付け、重複排除、スコア正規化
- Project層: `group_events` / ランキング集計 / キャッシュ更新
- Serve層: API + CDNキャッシュ + 検索インデックス

### 4.3 スケーラビリティ

- 読み取り高負荷は `read model` とキャッシュで吸収
- 重い集計は非同期ジョブへ隔離
- 失敗時は再試行キュー + dead letter queue で運用

## 5. ナンダッテ再設計（最重要）

### 5.1 仕様変更

- 投票対象を「自由ワード中心」から「テーマ + 候補 + 自由入力補助」に変更
- 自由入力は即時反映ではなく、正規化フローを通して統合
- 1人の投票履歴を可視化し、再投票や比較をしやすくする

### 5.2 不正対策

- レート制限（IP / device / account）
- 急激な同一パターン投票を自動隔離
- 通報導線 + モデレーションキュー

### 5.3 価値の再定義

- 順位を競うだけでなく、グループの魅力語彙を育てる辞書として運用
- 新規ファンにとって「最初に読むと魅力が分かるページ」を自動生成

## 6. 検索再設計（横断検索の中核化）

- 単一検索窓で `group / article / social_post / vote_topic` を同時検索
- 結果は「エンティティ別タブ + 時系列混在ビュー」の2モード
- サジェストは「正式名」「略称」「関連タグ」「最近急上昇」を混在表示
- 検索結果から直接 `Group Hub` に遷移し、離脱を減らす

## 7. 運用設計（開発・編集・分析）

### 7.1 開発プロセス

- Monorepo（app / jobs / shared schema / scripts）
- DBマイグレーションをPR必須化
- `staging` で夜間同期のリハーサルを毎日実施

### 7.2 編集運用

- ニュースはCMS編集、ランキング/バズ/投票は自動同期
- 例外修正は「手修正」ではなく「ルール修正」優先
- issueは「データ品質」「紐付け」「表示不整合」で分類

### 7.3 指標管理

- 体験KPI: 検索→詳細遷移率、詳細→投票率、7日再訪率
- 品質KPI: group紐付け精度、同期遅延、欠損率
- 事業KPI: DAU、アクティブ投票者数、グループ別参加密度

## 8. 実装フェーズ（90日想定）

### Phase 1（0-30日）基盤

- `groups` 中心スキーマ再構築
- `group_events` read model 導入
- 新検索API（最低限）

### Phase 2（31-60日）体験刷新

- `Group Hub` 新設
- Home / Explore / Participate を新導線へ移行
- ナンダッテの投票フロー改修（テーマ制）

### Phase 3（61-90日）最適化

- レコメンド（似たグループ、次に読む記事）
- 通知（フォローグループ更新）
- 分析ダッシュボードでKPI自動監視

## 9. リスクと対策

- リスク: 既存ユーザーが導線変更で迷う
- 対策: 旧UI導線を30-60日併存、段階リダイレクト

- リスク: group紐付け精度が低いと全体品質が崩れる
- 対策: ルールベース + 人手レビュー + 学習データ蓄積

- リスク: 投票の信頼性が疑われる
- 対策: 不正検知、透明な集計ルール、履歴表示

## 10. この再設計案の価値

- 情報の分散を構造的に解消できる
- 「読むサイト」から「参加して育つプロダクト」に進化できる
- 機能追加時も `Group` と `Event` に乗せるだけで拡張しやすい

## 11. GPT-5.4追記: ゼロベース再設計パターン比較

この章では、上記の「Group Timeline基盤案」を含めて、ゼロから再設計する場合の有力案を3つに分けて比較する。

### 11.1 案A: Group Timeline OS

- 中心思想: すべてを `Group` と `Event` に収束させる
- プロダクト像: 各グループに「今なにが起きているか」を最短で理解できる情報OS
- 主画面: Home / Explore / Group Hub / Participate / My Page
- データ設計: `groups` + `group_events` + 各ソースの正規化テーブル
- 強み: 情報統合に最も強く、拡張性が高い
- 弱み: 初期設計コストが高く、group紐付け精度が品質を左右する

### 11.2 案B: Search First Media

- 中心思想: サイト全体を「検索起点メディア」として設計する
- プロダクト像: ユーザーはまず検索し、結果からニュース・ランキング・話題投稿・投票へ横断移動する
- 主画面: Search Home / Search Results / Entity Detail / Topic Pages
- データ設計: 検索インデックス主導、アプリは検索結果を中心に構成
- 強み: 現在の「横断検索が強み」という特性を最も伸ばしやすい
- 弱み: 参加体験が従属機能になりやすく、コミュニティ性が弱い

### 11.3 案C: Fan Participation Network

- 中心思想: ナンダッテを核にして、投票・コメント・履歴・フォローを中心に再構成する
- プロダクト像: ファンがグループ像を共同編集していく参加型ネットワーク
- 主画面: Participate Home / Group Voting / Fan Boards / My Activity
- データ設計: `votes` `topics` `user_activity` を厚くし、ニュースやバズは補助情報として接続
- 強み: 差別化が最も明確で、熱量の高いコアファンに刺さりやすい
- 弱み: ライトユーザーには入口が狭くなり、ニュース基盤が弱いと広がりにくい

## 12. 評価軸

- 事業適合: IDOL CROSSINGの独自性を強くできるか
- 初期実装難易度: 90日以内に現実的に立ち上げられるか
- 運用安定性: 編集・同期・データ品質の運用が回るか
- 拡張性: 機能追加時に構造破綻しにくいか
- SEO/流入適性: 検索流入と記事導線を伸ばしやすいか
- 参加体験: ファンが「見るだけ」で終わらず参加したくなるか

評価は 5点満点。高いほど有利。

## 14. 結論

結論として、最も妥当なのは `案A: Group Timeline OS`。

理由:
- 現行の4機能を無理なく統合できる
- 「知る→見る→参加する」を一番自然に設計できる
- 将来、通知・推薦・比較・履歴・会員機能を足しやすい
- メディアと参加型サービスの中間ではなく、その両方を持てる

ただし、実務上は以下の折衷が最も現実的。

- 基盤は `案A`
- 集客導線は `案B` の Search First を部分採用
- 差別化の核として `案C` の参加体験を `ナンダッテ` に限定して強化

つまり、最終形は単一案ではなく、**Aを土台にBとCを部分統合する設計**が最も強い。

## 15. 推奨アクション

ゼロから本当に作り直すなら、設計順は以下がよい。

1. `Group` 中心のDB再設計を先に確定する
2. `group_events` を使った Group Hub を最初の主画面として作る
3. 横断検索を `Search First` で強化する
4. ナンダッテをテーマ制 + 不正対策込みで再設計する
5. 最後に通知、My Page、レコメンドを足す

この順番なら、基盤の破綻を防ぎつつ、早い段階で「新しくなった価値」が見える。

## 16. 追加観点: 弱者が強者に勝つ条件からの再評価

この章では、単なるプロダクト評価ではなく、以下の観点から再評価する。

- 競争戦略: 強者と同じ土俵で戦わず、どこで非対称を作れるか
- ゲーム理論: 相手の最適反応を前提にしても優位を維持できるか
- 進化論: 試行錯誤と学習速度で優位を作れる構造か
- 組織論: 少人数でも回り、大企業が模倣しにくい運営構造か

前提として、弱者が強者に勝つ条件は「資本量」ではなく、以下の構造を持てるかで決まる。

- 強者が本気で取りにくい小さいが濃い市場を押さえる
- ユーザー行動が増えるほどデータ優位とプロダクト優位が同時に積み上がる
- 模倣されても、運用密度とコミュニティ文脈で差が残る
- 意思決定速度と改善速度が、組織規模の不利を上回る

## 17. メタゲームの整理

IDOL CROSSINGが戦う相手は、単一の競合ではない。

- ニュース面では大手メディア
- 検索面ではGoogleやSNS内検索
- 話題検知ではXやYouTubeのアルゴリズム
- 参加面ではSNSコミュニティやファンダム文化そのもの

つまり、正面から「情報量」や「配信量」で勝つのは不利。勝ち筋は、**情報供給者になることではなく、ファン行動の編集装置になること**にある。

この意味で重要なのは、記事本数やDB件数ではなく、以下の3つ。

- どれだけファンの解釈を蓄積できるか
- どれだけグループ単位で文脈を圧縮できるか
- どれだけ参加行動を継続習慣に変えられるか

## 18. 各案の戦略再評価

### 18.1 案A: Group Timeline OS

競争戦略評価:
- 強者が持つ「大量配信」ではなく、「グループ単位の文脈統合」で差別化できる
- 大手が後追いしようとしても、全体最適よりカテゴリ横断統合が必要で、組織的に実装が重い

ゲーム理論評価:
- 競合がニュースを増やしてきても、こちらは「統合された理解」で差を維持できる
- 競合が検索を強化しても、Group Hubに履歴・参加・反応が蓄積すると単純模倣では追いつきにくい

進化論評価:
- データが `Group` に集約されるため、改善の学習サイクルが速い
- 検索、推薦、通知、投票の改善が同じ母集団データに乗るので進化効率が高い

組織論評価:
- 少人数でも「1つのコアモデル」を磨けば全体品質に効く
- 部門分業型の大組織より、一体運営の小チームが有利

戦略スコア:
- 非対称性: 5/5
- 模倣困難性: 4/5
- 学習速度優位: 5/5
- 組織適合性: 5/5

総評:
弱者の勝ち筋として最も強い。理由は、単機能勝負ではなく「統合の質」で戦えるから。強者が同じものを作れても、優先順位と組織構造の都合で後回しになりやすい。

### 18.2 案B: Search First Media

競争戦略評価:
- 流入は取りやすいが、強者も同じゲームをしやすい
- SEOや検索UIは、資本と編集体制のある組織が追随しやすい

ゲーム理論評価:
- 競合が同じキーワードを取りに来た時、防衛力が低い
- 最適反応をされた瞬間に、価格競争ならぬ記事量競争・SEO競争に入りやすい

進化論評価:
- 検索ログは溜まるが、参加行動が弱いため、学習が流入最適化に偏る
- 学習の結果が「より見つかりやすいメディア」に収束しやすく、独自進化しにくい

組織論評価:
- 編集運用には合うが、構造的にメディア組織へ引っ張られる
- 少人数チームでも回るが、結果として大手と同じ評価軸で戦いがち

戦略スコア:
- 非対称性: 2/5
- 模倣困難性: 2/5
- 学習速度優位: 3/5
- 組織適合性: 4/5

総評:
短期では合理的だが、強者に勝つ構造は弱い。これは「勝ちやすい」のではなく「参入しやすい」案であり、長期的には差別化が薄くなる。

### 18.3 案C: Fan Participation Network

競争戦略評価:
- 強者が最も取りづらいのは、濃い参加コミュニティ
- うまく回れば強烈な非対称が生まれる

ゲーム理論評価:
- 参加者が増えるほど退出コストと関係性資産が積み上がるため、防衛力は高い
- ただし初期人数が少ないとネットワーク効果が立ち上がらず、不成立リスクが高い

進化論評価:
- 参加ログ、語彙、投票傾向、コミュニティルールが蓄積すると非常に強い
- 一方で、誤進化も起こりやすく、ノイズや荒れを増幅すると逆進化する

組織論評価:
- 少人数運営ではモデレーション負荷が重い
- 組織能力がプロダクト力と同じくらい重要になり、設計だけでは勝てない

戦略スコア:
- 非対称性: 5/5
- 模倣困難性: 5/5
- 学習速度優位: 4/5
- 組織適合性: 2/5

総評:
当たれば最強だが、再現性が低い。弱者が勝つ条件のひとつである「強者が真似しにくい」は満たすが、同時に「自分たちが安定運営できる」条件を満たしにくい。

## 19. 戦略コンサル視点での結論

結論は前章と同じで、基盤は `案A: Group Timeline OS` が最善。ただし理由は「バランスが良いから」ではなく、**弱者の勝ち筋に最も整合するから**である。

その理由は以下。

- 強者と同じKPIゲームに入らずに済む
- ユーザー行動が増えるほど、統合品質と参加品質の両方が上がる
- 小チームの高速改善が、そのまま競争優位になる
- 大手が模倣しても、組織構造上の後追いになりやすい

一方で `案B` は強者の得意ゲームに近く、`案C` は勝てる時は大きいが運営能力の要求が高すぎる。したがって、勝率で見れば以下の並びになる。

1. `案A` を基盤にする
2. `案A` に `案C` の参加蓄積を慎重に注入する
3. `案B` は集客レイヤーとして限定利用する

## 20. 最終再定義

この観点を踏まえると、IDOL CROSSINGの本質は「アイドル情報サイト」ではない。

より正確には、**ファンの解釈・反応・参加を、グループ単位で蓄積して再利用できる知識基盤**である。

この定義に立つと、勝ち筋は明確になる。

- ニュースを増やすことではなく、文脈を圧縮する
- 検索を作ることではなく、次の行動を誘発する
- 投票を置くことではなく、参加の履歴を資産化する

この方向なら、弱者でも強者に対して局所優位ではなく構造優位を作れる。

## 21. 現行版と再設計版の並行は可能か

可能。むしろ現時点では、その進め方が最も合理的。

理由:
- 現行版はまだ評価母数が足りず、止めて全面再設計に振るのは早い
- 一方で、再設計の探索を止めると、中長期の勝ち筋の学習が遅れる
- よって、「現行改善」と「次世代設計の研究開発」を分離して並走させるのがよい

ただし、同時並行が成立するのは、両者の責務を明確に分けた場合に限る。

## 22. 並行の基本原則

### 22.1 二層で運営する

- 現行トラック: 現在のデプロイ版を改善し、実ユーザーの反応を集める
- 再設計トラック: 次世代構造を小さく検証し、将来の本命設計を育てる

### 22.0 実装前提の制約

- 現行版に影響が出る作業は行わない
- 既存DBのスキーマ変更は行わない
- 既存APIや既存画面の挙動変更を前提にしない
- 再設計トラックは、現行版から論理的にも物理的にも分離して進める
- 設計と実装の優先順位は、都度ユーザーが細かく決めるのではなく、GPTが最適と判断する順序を基本とする

この制約により、再設計トラックは「現行版の改修計画」ではなく、「隔離された次世代プロトタイプ計画」として扱う。

### 22.2 混ぜる場所を限定する

共有してよいもの:
- `group` の識別子
- 計測イベント定義
- 一部の正規化ルール
- 将来使い回せるデータ基盤

共有しないほうがよいもの:
- 画面構成
- 導線思想
- UIコンポーネント設計
- ページ単位の情報設計

### 22.3 現行版を実験場にしすぎない

- 現行版では、改善はしても設計思想の大転換はしない
- 再設計思想は、独立トラックで検証してから逆輸入する

## 23. 実務での進め方

### 23.1 現行トラックでやること

目的:
- 今のユーザーがどこに反応するかを把握する
- 何が刺さり、何が迷われ、何が使われないかを知る

やること:
- 主要導線の計測追加
- ホーム、検索、詳細、投票の離脱率確認
- 現行機能の改善を小さく速く回す
- グループ詳細やランキング詳細など、今ある体験を磨く

見たい指標:
- 検索利用率
- グループ詳細遷移率
- 詳細から投票への遷移率
- 再訪率
- よく見られるグループと見られないグループの差

### 23.2 再設計トラックでやること

目的:
- `Group Timeline OS` の成立条件を検証する
- 現行UIの延長では作れない体験を試す

実装条件:
- 既存DBは読取り専用で扱うか、必要ならスナップショット/別DBへ複製して使う
- 新テーブル、新API、新画面は再設計トラック側に閉じ込める
- 既存コードの置換ではなく、独立実装を原則にする

最初に作るべきもの:
- `groups` 中心の共通スキーマ案
- `group_events` の試作
- `Group Hub` のプロトタイプ

この段階では、Home全体や全機能の置き換えはやらない。
最小単位で十分。

最初の検証対象:
- 1グループの文脈統合が本当に便利か
- ニュース、ランキング、バズ、投票を一画面でつなぐ価値があるか
- 参加履歴があると継続率が上がるか

### 23.3 合流ポイントを先に決める

並行推進で重要なのは、「どこで合流できるか」を先に固定すること。

合流候補:
- `group` マスタ
- イベント計測
- 検索インデックス用の正規化データ
- `group_events` に必要なソースデータ

逆に、すぐ合流させないもの:
- 新UI全体
- My Pageの新設計
- 参加機能の全面刷新

## 24. 推奨ロードマップ

### Phase A: 現行版の学習期間

- 期間目安: 1-2か月
- 現行版を運用しながらKPI観測を行う
- 同時に、再設計トラックでは `group` 中心のデータモデルだけ設計する
- 再設計側の実装は、別DBまたはローカル/検証用データセット上でのみ行う

完了条件:
- 現行版の主要導線に十分な観測データが溜まる
- `group` マスタとイベント定義の叩き台が固まる

### Phase B: 再設計の最小試作

- 期間目安: 次の1か月
- 独立トラックで `Group Hub` を試作する
- 対象は限定された数グループでよい
- 内部検証または限定公開で評価する
- 現行版の本番系データ更新を伴わない形で完結させる

完了条件:
- 一画面統合の価値が確認できる
- 現行版に逆輸入できる要素が見える

### Phase C: 逆輸入

- 再設計版の成果を現行版に部分導入する
- たとえば以下は逆輸入しやすい
- グループ文脈カード
- 時系列要約
- 参加履歴の見せ方
- 検索サジェスト改善

完了条件:
- 現行版の改善と次世代設計が矛盾しなくなる
- 将来の本移行に向けた共通土台が育つ

## 25. 実際の運用ルール

### 25.1 ブランチとタスク管理

- 現行改善は通常ブランチで運用する
- 再設計は `redesign/` 系ブランチや独立ディレクトリで隔離する
- issue も `current` と `redesign` に分ける

### 25.2 毎週の見方

- 現行トラックでは「数字がどう動いたか」を見る
- 再設計トラックでは「仮説が成立したか」を見る

この2つを同じ会議で混ぜると判断が鈍る。
レビューの場も分けたほうがよい。

### 25.3 意思決定ルール

- 現行版に入れる変更は、既存KPI改善に効くものだけ
- 再設計で作るものは、将来の勝ち筋検証に効くものだけ
- 「なんとなく良さそう」は両トラックで禁止
- 再設計トラックの設計優先順位は、毎回人手で細かく決めず、GPTが最適と判断する順に進める
- ただし、現行版への影響が発生する提案は、優先度に関係なく不採用とする

## 26. 避けるべき進め方

- 現行版を止めて全面再設計に入る
- 現行UIに再設計思想を中途半端に混ぜる
- 共通データモデルを決めずに並行開発する
- 再設計側で機能を作りすぎて、検証より実装が先行する
- 現行版と同じDBに再設計用テーブルを安易に足す
- 現行版で使っているスキーマを、将来のために先回り変更する

特に危険なのは、現行版の改善なのか再設計の試作なのかが曖昧になること。
これが起きると、両方とも中途半端になる。

## 27. 結論

並行は可能であり、今はその進め方が最適。

ただし形としては「2つのプロジェクトを同時に作る」のではなく、以下の構造で進めるべき。

- 現行版: 市場学習装置
- 再設計版: 勝ち筋探索装置
- 共通基盤: 将来の合流装置

この3層で整理すると、今のプロダクトを止めずに、中長期の本命設計も育てられる。

## 28. この制約下での実装方針

この前提でGPTが優先すべき実装順は、以下になる。

1. 現行版に触れない独立データモデルを定義する
2. そのモデルに載せるための最小データ投入パイプラインを作る
3. `Group Hub` の最小UIを独立画面として試作する
4. 計測と検証方法を定義する
5. 有効だった要素だけを、将来の統合候補として整理する

この順が最適な理由:
- 現行版への影響ゼロを守れる
- 設計の良し悪しを、言葉ではなく実装で判断できる
- いきなり全面再設計せず、中核仮説だけを検証できる

つまり、次にやるべきことは「既存環境を触らずに動く、小さな再設計実装の土台作り」である。
