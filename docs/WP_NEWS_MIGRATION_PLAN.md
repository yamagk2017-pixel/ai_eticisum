# WordPressニュースメディア移行方針メモ（ヘッドレスWP + Sanity）

## 目的

既存のWordPressニュースメディア（約3,500件の記事）を、このプロジェクト（Next.js）へ段階的に移行する。

前提:
- 記事は今後も増える
- 既存WPはすぐには閉鎖しない
- 将来的にはWPを閉鎖する可能性が高い
- WPはクラシックエディタのみ使用
- 編集者は1人（本人）

## 現時点の基本方針（合意）

段階移行で進める。

1. 既存記事は `ヘッドレスWP` として取り込み（まずは表示/UI最適化）
2. 今後の新規記事は `Sanity` で投稿
3. 既存WP記事も優先度順に `Sanity` へ徐々に移行
4. SEO対策（canonical / 301など）は別途設計して進める

この方針の意図:
- 一括移行の初期コストを下げる
- UI刷新を先に進められる
- WPを原本として残しながら安全に移行できる
- 中長期でSanity中心運用へ寄せやすい

## ヘッドレスWPの理解（整理）

ヘッドレスWP = `WordPressをCMS/記事保管庫として使い、表示はNext.js側で行う` 方式。

今回の理解:
- Next.js側から WP REST API を呼んで記事を取得
- 取得したデータをこのサイトのUIで表示
- WPテーマの見た目は基本使わない（または最小限）

### 「記事が両方に存在するのか？」について

構成次第で2パターンある。

- 純ヘッドレス（参照のみ）
  - 記事実体はWPのみ
  - Next.jsはAPIで読んで表示するだけ
- 同期あり（WP + 別DB）
  - WPを原本にしつつ、別DBにも複製保持する
  - 高速表示・検索・集計・差分同期管理に使える

今回のMVP段階では `純ヘッドレス（参照のみ）` で十分。

## Supabaseの位置づけ（今回の結論）

### MVPでは必須ではない

「最新記事1件を表示して確認する」だけなら `Supabaseなし` で実装可能。

### 将来的に使う理由（必要になったら）

Supabaseは主に以下の用途:
- WP APIのキャッシュ（高速化）
- 差分同期の管理（新規/更新/失敗リトライ）
- 検索・集計・ランキング
- WPにない独自データの保存
- 将来の完全移行の足場

## Sanity採用方針（現時点）

Sanityは有力。特に以下の条件で相性が良い:
- 編集者が1人
- クラシックエディタ運用（Gutenberg変換問題が小さい）
- 今後の構造化（SEO項目・関連記事・特集など）を強化したい

### 実務的な使い分け（推奨）

- 既存記事: 当面はヘッドレスWPで表示
- 新規記事: Sanityで投稿開始
- 旧記事: 人気記事/重要記事から段階的にSanityへ移行

## WP APIの状態確認（実施済み）

以下を確認済み:
- `https://<WPドメイン>/wp-json/wp/v2/posts` が応答
- JSON（記事配列）が返る

これにより、少なくとも以下は満たしている:
- WP REST APIが有効
- 公開記事の取得が可能
- ヘッドレス化の入口は問題なし

## WP APIアクセス時の考え方（整理）

### 推奨構成

`ブラウザ -> Next.js -> WP API`

理由:
- CORS問題を回避しやすい
- 認証情報をクライアントに出さない
- キャッシュしやすい
- 後でSanity/Supabase導入しても差し替えしやすい

### WP側で必要になりやすい確認事項

- `/wp-json/` が開くか
- `/wp-json/wp/v2/posts` が取得できるか
- セキュリティプラグイン/WAF/CloudflareがREST APIを塞いでいないか

## MVP実装（このリポジトリ内・実施済み）

目的:
- Supabaseなしで、WPの最新記事1件を表示確認する

追加したファイル:
- `src/lib/wp/client.ts`
- `src/app/wp-mvp/page.tsx`

### 実装内容

- `WP_API_BASE_URL` を元に WP REST API を呼ぶ
- `?per_page=1&_embed` で最新記事1件 + アイキャッチを取得
- タイトル / 抜粋 / 本文 / アイキャッチを表示
- 環境変数未設定時は画面上にセットアップ案内を表示

### 環境変数

`.env.local` に以下を設定:

```env
WP_API_BASE_URL=https://your-wordpress-site.example.com
```

補足:
- 末尾 `/` はあってもなくてもOK（実装側で吸収）
- `/wp-json/wp/v2/posts` まで含めない（ベースURLのみ）

### 確認URL

- `/wp-mvp`

## UI/デザインの自由度（結論）

かなり自由にカスタマイズ可能。

### 自由に作れる部分（Next.js側）

- 一覧カードUI
- 記事詳細レイアウト
- タイポグラフィ/余白/配色
- 関連記事/CTA/導線
- モバイル最適化

### WP依存が残りやすい部分

- `content.rendered` のHTML構造
- 埋め込みコード
- ショートコード
- WPプラグイン依存のHTML/CSS/JS

実務的には:
- まずは記事ページの器/UIを作る
- 次に本文HTML向けスタイルを整える
- 必要に応じてHTML変換レイヤーを入れる

## フォトギャラリープラグインについて（要確認）

ギャラリーは `そのまま動く場合` と `崩れる場合` がある。

難易度は `content.rendered` に何が返ってくるかで決まる。

### 想定パターン

1. 完成HTMLが返る
- 比較的対応しやすい
- Next側でCSSを当てれば表示可能なケースが多い

2. ショートコードのまま返る
- そのままでは表示崩れ/文字列表示の可能性
- 変換処理が必要

3. JS/CSS依存
- WPテーマ/プラグインのJS/CSSがないとスライダー等が動かない
- Next側で代替コンポーネント化が必要なことがある

### 次の確認方法

ギャラリー入り記事の `content.rendered` を確認し、以下を判定する:
- HTML化済みか
- ショートコード残りか
- JS依存が強いか

## 画像ファイルの方針（重要）

認識は正しい。将来WPを閉鎖するなら、画像を永続的にWP参照し続けるのは避けるべき。

### 推奨方針（段階的）

#### Phase 1（短期）
- WP画像URLをそのまま参照
- まず表示/導線/UIを作る

#### Phase 2（移行基盤）
- 画像保存先を決める
  - `Sanity Assets`
  - `Supabase Storage`
  - `S3 / Cloudflare R2` など
- 画像移行スクリプトの方針を決める

#### Phase 3（段階移行）
- 新規Sanity記事は新保存先に直接アップロード
- 旧WP記事は優先度順に画像移行
- 本文内画像URLの置換

#### Phase 4（WP閉鎖前）
- WP画像参照の残件棚卸し
- URL置換漏れ/表示崩れ確認
- WP画像参照ゼロを目標にする

### 先に設計しておくべき事項

- `WP画像URL -> 新URL` の対応表の管理方法
- 本文内URL置換ルール
- ファイル名重複対策（ハッシュ or 年月パス維持）
- `alt` / caption の保持方針

## SEOに関する注意（未設計・要対応）

ヘッドレス化できることと、SEO的に安全に公開できることは別問題。

重複公開期間は特に注意:
- 旧WPサイトと新サイトで同一記事を同時公開すると重複扱いの可能性

要検討:
- canonicalの正本ルール
- 301移行のタイミング
- 既存記事をどの単位で新サイト公開するか（全件/カテゴリ単位/段階的）

## 決定事項（現時点）

- 段階移行で進める
- 既存記事は当面ヘッドレスWPで取り込む
- 新規記事はSanity中心を検討（実質採用候補）
- 旧記事は徐々にSanityへ移行
- MVPはSupabaseなしで進める
- 画像は短期WP参照 / 中長期で移行する前提

## 未決定事項（要判断）

- Sanityの正式採用タイミング（いつから新規投稿を切り替えるか）
- 記事URL設計（WP由来/Sanity由来の統一方針）
- SEO運用ルール（canonical / 301 / 公開順）
- 画像保存先（Sanity Assets / Supabase Storage / R2等）
- ギャラリープラグイン記事の扱い（そのまま / 変換 / 代替UI）
- 本文HTMLの整形方針（どこまで自動変換するか）

## 直近の次アクション（おすすめ）

- `wp-mvp` を土台に、記事詳細の見た目を整える

2. `一覧（最新5件など）を作る`

3. `ギャラリー入り記事のAPIレスポンス確認`

4. `Sanityの最小スキーマを作る`
- Article / Category / Tag / SEO（まずはHTML本文対応でも可）

5. `SEO方針メモを別紙で作る`
- canonical/301/公開移行ルールを先に決める
