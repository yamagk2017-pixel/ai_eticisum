# トップページ表示速度 改善メモ（2026-03-13時点）

## 1. 現状サマリー
- トップページは ISR 化済み（`revalidate=300`）で、毎リクエストのフル再集計は止めた。
- `getHomeSummaries` は `unstable_cache` でキャッシュしている。
- ニュース取得は `limit=12` まで削減済み。
- ナンダッテはトップ内でクライアントポーリング化し、`/api/nandatte/rankings` を `no-store` で取得する構成にした。

## 2. まだ遅く感じる主な要因（仮説）
- ホームカード自体が多く、初期HTML量と描画量が大きい。
- バズッタラの埋め込み（`react-tweet`）は後段読み込みでも、体感上の重さ要因になりやすい。
- トップカードの一部は画像が多く、ネットワークとレイアウト計算コストが高い。
- ISR再生成タイミングでは外部依存（Supabase/WP/Sanity）待ちが長くなる可能性がある。
- 「初期表示」と「表示後更新」が混在し、ユーザー体感では「遅い」に見えやすい。

## 3. 優先改善案（短期）
1. Above-the-fold 最適化
- ファーストビューに必要なカードだけを先に出し、残りは遅延表示する。
- `buzzttara` 埋め込み領域は初回は軽量プレースホルダ優先にする。

2. 画像最適化
- 重要画像以外はサイズを明示し、`loading="lazy"` と `sizes` を厳密化する。
- 画像URLの解像度が過剰な場合はサムネイル利用に寄せる。

3. ナンダッテの通信制御
- ポーリング間隔を 15秒固定から可変（例: 15/30/60秒）にする。
- 非アクティブタブ時はポーリング停止または間引きする。

## 4. 中期改善案
1. トップカードの分割配信
- 重いカードを Server Components 単位で分割し、段階的レンダリングを強化する。

2. ニュース集約処理の軽量化
- 「カードに必要な最小フィールドのみ取得」へ寄せる。
- カテゴリ別に必要最小件数だけ取得し、全件横断を減らす。

3. キャッシュ無効化戦略の整理
- 更新トリガーが取れるデータは `revalidateTag` ベースの明示無効化を検討する。

## 5. 計測と運用ルール
- 毎週、以下を記録する。
  - LCP（モバイル）
  - INP
  - TTFB
  - ホーム再生成時間（ISR再検証時）
- 計測は最低 3 回の中央値を採用する。
- 施策ごとに「体感」「実測」「運用コスト」をセットで評価する。

## 6. 直近の実行順（推奨）
1. ファーストビュー以外カードの遅延表示
2. バズッタラ埋め込みの軽量プレースホルダ化
3. ナンダッテのポーリング最適化（タブ非アクティブ時の間引き）
4. ニュース取得フィールドの最小化
